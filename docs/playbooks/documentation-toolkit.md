# Playbook: инструментарий документации (Markdown)

## 1. Цель

Дать команде простой и понятный процесс работы с документацией:

- писать документы в Markdown;
- проверять, что ссылки и структура не сломаны;
- собирать локальные версии в PDF/DOCX для передачи заказчику.

## 2. Важное правило для GitHub

**PDF и DOCX не храним в репозитории.**

Причина: это бинарные файлы, они быстро «засоряют» историю Git и усложняют review.
В GitHub держим только исходники (`.md`) и скрипты сборки.

## 3. Набор инструментов

- `scripts/export-docs.sh` — сборка из Markdown в DOCX и PDF (локально).
- `docs/guides/documentation-export-pdf-docx.md` — инструкция для пользователя инструмента.
- (опционально) `pandoc` + один PDF-движок (`wkhtmltopdf`, `weasyprint` или `xelatex`).

## 4. Базовый процесс

1. Обновить/добавить Markdown-документы.
2. Проверить понятность текста (без перегрузки узкой терминологией).
3. Запустить экспорт локально:

```bash
./scripts/export-docs.sh
```

4. Передать PDF/DOCX заказчику любым внешним каналом (файлообменник, почта, внутренний портал),
   **но не добавлять их в git**.

## 5. Что обязательно для документации

- Русский язык.
- Простая подача: текст должен быть понятен специалисту с базовым/средним техническим уровнем.
- Короткие шаги «что делать» вместо длинных теоретических блоков.
- Разделы для ролей: пользователь, внедренец, поддержка, продажи.

## 6. Definition of Done для документационных задач

- Есть обновлённый Markdown-файл.
- Есть ссылка на него в `README.md` (если документ новый).
- Экспортный скрипт не сломан (`bash -n scripts/export-docs.sh`).
- В PR нет PDF/DOCX-бинарников.
