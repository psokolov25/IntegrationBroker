# Playbook: стандарты README.md

## 1. Цель

Сделать README понятным для трёх аудиторий:

- пользователя/заказчика;
- внедренца/поддержки;
- разработчика.

README должен помогать быстро ответить на вопросы: «что это?», «как запустить?», «где документация?», «как эксплуатировать безопасно?».

## 2. Рекомендуемая структура README

1. **Название проекта**
2. **Бейджи** (технологии, сборка, документация)
3. **Короткое описание** (2–4 строки)
4. **Содержание** (TOC)
5. **Назначение / scope**
6. **Архитектура** (крупные блоки)
7. **Быстрый старт** (сборка, запуск, проверка)
8. **Конфигурация / окружение**
9. **Документация и playbooks**
10. **Эксплуатация и безопасность**
11. **Лицензирование / вклад**

## 3. Бейджи: что использовать

Минимальный набор:

- версия платформы (например, Java 21);
- framework/runtime;
- тип сборки (Maven/Gradle);
- политика документации (например, `markdown-first`).

Правила:

- не добавлять «шумные» бейджи без пользы;
- не добавлять нестабильные бейджи, которые часто ломаются;
- все бейджи должны быть читаемы в светлой и тёмной теме.

## 4. Контент-стандарты

- Писать на русском языке (простыми формулировками).
- Сначала практические шаги, потом детали.
- Избегать длинных абзацев — лучше короткие списки и чек-листы.
- Любые команды должны быть копируемы «как есть».
- Для путей и endpoint использовать моноширинный формат (`` `...` ``).
- Если добавлен новый документ — обязательно добавить ссылку на него в README.

## 5. Чек-лист перед merge

- [ ] README содержит актуальные разделы и TOC.
- [ ] Быстрый старт запускается по командам из README.
- [ ] Ссылки на playbooks/guides не битые.
- [ ] Нет бинарных файлов в документации.
- [ ] Раздел безопасности отражает текущие ограничения (секреты, корреляция, доступы).

## 6. Анти-паттерны

- «Полотно» текста без структуры.
- Неактуальные команды запуска.
- Ссылки на удалённые/переименованные документы.
- Избыточные внутренние термины без пояснений.
- Перечень «всего подряд» без приоритета для пользователя.
